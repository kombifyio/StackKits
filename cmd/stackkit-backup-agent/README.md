# stackkit-backup-agent

Per-host orchestrator for the kombify-Backup SaaS path.

## What it does (when complete)

1. Authenticates against the kombify-Backup-Controller using a per-host
   token issued at enrollment time.
2. Pulls JobMessages from the controller's queue.
3. Executes them by exec'ing the local `kopia` client (the same binary
   the `addons/backup` add-on already runs).
4. Reports status back to the controller.

## What it does today (scaffold)

This binary ships as a Phase-4 scaffold. It accepts the planned CLI
surface (`enroll`, `run`, `status`) and validates flags, but exits with
a clear "not implemented" message because the controller's network
listener is not yet stood up. See
[`docs/plans/2026-05-01-backup-rollout.md`](../../docs/plans/2026-05-01-backup-rollout.md)
for status.

The agent is **not required** for self-hosted users. The local
`kopia-agent` container plus the Kopia Web UI on `https://backups.<domain>`
is the complete self-hosted surface; this binary only matters when the
host is part of a multi-tenant fleet driven by the controller.

## Usage (planned)

```bash
# One-time, copy the token from the kombify-TechStack dashboard:
stackkit-backup-agent enroll \
  --token  $AGENT_TOKEN \
  --endpoint https://backup.kombify.io

# As a long-running service (will be wrapped by a systemd unit
# generated by the addon when agentMode is enabled):
stackkit-backup-agent run --config /etc/stackkit/backup-agent.yaml

# One-shot status check:
stackkit-backup-agent status
```

## Distribution

Linux-only (amd64 + arm64), shipped as a standalone tarball via
goreleaser. Operators install one of these on each host of a fleet,
alongside their existing StackKit deployment.
